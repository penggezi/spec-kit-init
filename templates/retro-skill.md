## 模式选择

根据 $ARGUMENTS 决定走哪条路径：

| 输入 | 模式 | 触发场景 |
|------|------|----------|
| 空 | 标准复盘 | /speckit-implement 完成后自动询问、或你随时输 /retro |
| capture | 纠正捕获 | 你刚指出 AI 的错误或做了纠正（AI 主动调用） |
| 其他文本 | 焦点复盘 | /retro 刚才那个 bug 把输入作为本次复盘的范围限定 |


## 设计目标

让"做完一件事"和"留下经验"成为一个不会被遗忘的动作。每条存入的经验都经过系统化筛选，确保 `lessons.md` 中每一条都值得下次会话阅读。

- **自动触发**：`/speckit-implement` 完成后主动询问"是否复盘"
- **手动触发**：任何时候输 `/retro`
- **纠正捕获**：你指出 AI 错误时，AI 应主动调用 `/retro capture`
- **质量优先**：宁可漏掉一条真经验，也不要存入一条噪音
- **参考源驱动**：经验不是对"做了什么"的总结，而是对"参考了什么、参考漏了什么/错了什么"的勘误

## 标准复盘流程

### 1. 收集素材

不再单纯依赖会话上下文回忆，而是**以"本次工作参考了什么"为线索**来挖掘经验。核心原则：

> 经验 = 参考源的勘误/补充。每条经验本质上是对某个参考文档的"它没说清楚"或"它说错了"的记录，让下次会话读到同一参考源时能少踩一次坑。

#### 1.1 识别参考源清单

基于当前会话上下文，列出本次工作实际参考过的所有来源：

| 类型 | 来源举例 | 收集方式 |
|------|---------|---------|
| **项目指令文件** | CLAUDE.md、AGENTS.md、GEMINI.md | 会话中 Read 过的项目根文件 |
| **经验库** | lessons.md、constitution.md、memory 文件 | 会话中 Read 过的 .specify/memory/ 下文件 |
| **现有代码** | 项目中的模块、配置、API 定义 | 会话中 Read 过的源码文件 |
| **Skill 内容** | TDD skill、code-review skill 等 | 会话中通过 Skill 工具加载过的 skill |
| **模板/约定文件** | spec、plan、tasks 模板 | 会话中 Read 过的 spec-kit 模板 |
| **外部文档** | 官方文档、博客、issue、StackOverflow | 会话中通过 WebFetch/WebSearch 获取的内容 |
| **依赖库的源码/类型定义** | package.json、类型声明、SDK 文档 | 会话中 Read 过的 node_modules 或依赖配置 |

如果会话很短、没有参考过任何实质内容，**直接告诉用户"本次会话尚无值得沉淀的经验"** 然后退出，不要硬编。

#### 1.2 逐参考源分析

对每个参考源，逐一回答以下 4 个问题：

| 问题 | 含义 | 产出 |
|------|------|------|
| **准确吗？** | 该参考源所述内容与当前项目的实际行为是否一致？ | 发现"文档说错了"的勘误 |
| **完整吗？** | 该参考源是否遗漏了关键前提、版本差异、边界情况？ | 发现"文档没说的坑"的补充 |
| **适用吗？** | 该参考源给出的通用建议在当前项目/技术栈/版本下是否需要调整？ | 发现"通用建议不适用此场景"的适配说明 |
| **费解吗？** | 是否有花了大量时间才从模糊表述中理解清楚的要点？ | 发现"别人读这里也会卡住"的解读 |

**只要 4 个问题中任何一个有发现，就产出一条候选经验。**

#### 1.3 形成候选经验清单

每条候选经验按以下结构组织（尚未写入，先列在候选清单中）：

```
参考源: {文件路径/URL/Skill名}
参考源说: "{原文摘要或概括}"
实际上: "{在当前项目中遇到的反例或不足}"
教训: "{下次遇到类似场景应该怎么做}"
```

进入下一步质量门禁筛选。

### 2. 5 层质量门禁

每一条候选经验依次过筛，**任何一关不过即丢弃**：

| 关卡 | 问题 | 丢弃什么 |
|------|------|----------|
| 1 可复用性 | 这条对参考源的勘误/补充在未来类似场景中有用吗？ | 一次性场景、仅本次有效的临时发现 |
| 2 非平凡性 | 这个参考源的问题是"需要实际踩坑才能发现的"，还是"看一眼就能注意到的"？ | 参考源中显而易见的笔误、排版错误；真正有价值的是隐蔽的误导、遗漏的前提、版本差异 |
| 3 类型适配 | 它能明确归到哪个桶？（规范/约定/质量/业务） | 连分类都说不清的模糊观察 |
| 4 可验证性 | 参考源与实际的差距是否在本次工作中被实际验证过了？ | 只是"感觉文档写得不好"但未实际证实的主观判断 |
| 5 显著性阈值 | 这条经验如果丢失，下次会话读到同一参考源时会踩同样的坑吗？ | 丢了也无所谓、下次读文档会自动注意到的内容 |

**通过标准**：必须同时通过 ①-④，且第 ⑤ 关回答"是"。

### 3. 对抗审查

每条通过门禁的经验，再经过两个独立角色的审查。

#### 执行方式

**子代理环境（Claude Code）**：分别 spawning 两个子代理。

1. 读取 `references/mechanism-auditor.md`，将其内容作为机制审计员子代理的 prompt
2. 读取 `references/routing-auditor.md`，将其内容作为路由审核员子代理的 prompt
3. 将候选经验作为待审查内容传给每个子代理
4. 两个角色各自独立判断，都通过才放行

**无子代理环境（Codex / Copilot / Cursor 等）**：在 prompt 内分两步执行。

1. 读取 `references/mechanism-auditor.md`，按其中的审查标准判断当前经验
2. 在两步之间清空判断状态，换一个角色重新审视
3. 读取 `references/routing-auditor.md`，按其中的标准做路由判断

> 如果 `references/` 下的文件不存在（如 skill 安装不完整），退回到内联的简化版判断标准：机制审计 → 经验是否解释了「为什么会这样」的因果链条（而非仅「不要做 X」）；路由审核 → 违反后「破坏架构」归 constitution、「多花时间」归 lessons。

---

### 4. 去重（基于 lessons.idx）

写入前，读取 `.specify/memory/lessons.idx`（如文件不存在则跳过此步）。

`lessons.idx` 是一个独立的轻量索引文件，每行一条记录，格式为：

```
YYYY-MM-DD · 根因关键词 · 简述
```

**读取方式**：先只读 `lessons.idx` 做根因关键词比对，命中时才 Read `lessons.md` 对应正文做精确比对。索引文件极轻量，确保去重扫描不随经验积累而膨胀。

比对规则：不是看标题或文字是否相似，而是比对根因+解法组合：

| 比对结果 | 处理 |
|----------|------|
| 全新经验（根因+解法均不匹配） | 正常写入 |
| 高度重复（根因+解法高度一致） | 跳过写入，可告知用户该经验已有记录，跳过重复 |
| 相似但不同（相同根因，不同解法） | 标记为冲突条目，写入时带冲突标记，让用户下次裁决 |


### 5. 二路分流 + 用户确认

将通过门禁+审查+去重后的经验按路由审核员的结论分桶，输出预览清单：

```
## 本次复盘预览

### B. lessons.md (项目经验) — 3 条
- 事务回滚陷阱: Spring @Transactional 中 try-catch 吞掉异常导致不回滚...
- 分布式锁范围: 锁只覆盖 MODIFYING 状态，首次并发请求双写...
- 幂等处理: 所有对外 RPC 调用必须处理幂等

确认写入？(全部 / 选择性 / 跳过)
```

等用户回应再写入。**不要预先把"无内容"的桶硬填**——某一类没素材时直接省略该桶。

### 6. 写入

按用户确认结果执行：

- **A. constitution.md**：将宪章修订建议写入 `.specify/memory/constitution.md`，标明"待评审"状态
- **B. lessons.md**：追加到 `.specify/memory/lessons.md` 顶部（最新在上），同时往 `lessons.idx` 追加一行索引记录

### 7. 最终总结

```
✓ 已沉淀 X 条经验（门禁过滤 Y 条，去重跳过 Z 条）
- constitution.md: N 处修订
- lessons.md: M 条

建议提交: docs: 沉淀 XX 经验（来自 YYYY-MM-DD 复盘）
```


## 纠正捕获模式（/retro capture）

### 触发条件

此模式由 **AI 主动调用**（不是用户调用）。当以下情况发生时，AI 应主动调用 `/retro capture`：

- 用户说"不对"、"错了"、"应该是 X 不是 Y"
- 用户说"这段代码不应该这么写"、"这个做法有问题"
- 用户说"记住，以后要用 X 方法"
- 用户纠正了 AI 的输出、方案、代码或理解

### 执行流程

#### 1. 定位纠正内容

基于当前会话上下文，找到刚才被纠正的那件事，提取：

- 错误是什么：AI 做了什么/说了什么被纠正
- 正确做法：用户指出的正确方式是什么
- 发生场景：在什么上下文中发生的（哪个文件、哪个功能、哪类问题）

如果找不到明确的纠正点（会话已经过了很久），直接退出，告诉用户未找到最近的纠正点，请使用标准复盘手动沉淀。

#### 2. 质量门禁（轻量版）

只过三关：

| 关卡 | 问题 |
|------|------|
| 1 可复用性 | 这条纠正对其他场景也有价值吗？ |
| 3 类型适配 | 能分到哪个桶？ |
| 5 显著性阈值 | 丢了会出问题吗？ |

不通过则丢弃，不需要告诉用户（这只是一次纠正，不是正式复盘，无价值就不打扰）。

#### 3. 对抗审查

执行与标准复盘相同的对抗审查（机制审计+路由审核），但不单独 spawning 子代理——因为是轻量模式，读取 `references/mechanism-auditor.md` 和 `references/routing-auditor.md` 后在 prompt 内分步完成即可。

#### 4. 去重（基于 lessons.idx）

读取 `.specify/memory/lessons.idx` 检查是否已有相似的根因条目。重复则静默跳过（不追问用户）。

#### 5. 确认与写入

根据经验类型决定是否需要用户确认：

| 类型 | 处理方式 |
|------|----------|
| 非业务类（规范、约定、质量模式） | 直接写入，可一句话告知用户 |
| 业务类（业务规则、领域知识） | 需用户确认：输出一行简洁确认，用户点头再写入 |

判断标准示例：
- 非业务（自动写入）：DAO 层应该用 JdbcTemplate、public 方法必须写文档
- 业务（需确认）：这个订单状态机不支持逆向流转、退款金额上限是 5000

写入 `lessons.md` 顶部，同时往 `lessons.idx` 追加一行索引记录。一句话告知用户结果。


## 焦点复盘模式

当 $ARGUMENTS 不是空也不是 capture 时，把它作为本次复盘的范围限定。流程与标准复盘一致，但在素材收集阶段只关注与输入范围相关的参考源。

例如 `/retro 刚才那个 bug`——只分析为解决该 bug 而查阅过的参考文档、阅读过的相关代码，忽略同一会话中无关的其他参考源。

## 文件格式约定

### lessons.md 条目格式

```markdown
## YYYY-MM-DD · {简短标题}

**参考源**: {文件路径/URL/Skill名} — 本次工作读过的文档或代码，经验由此而生

**参考源说**: "{原文摘要或概括——文档/代码告诉了你什么}"

**实际上**: "{在当前项目中遇到的反例、遗漏的前提、版本差异或需要额外理解才能用对的要点}"

**教训**: "{下次遇到类似场景应该怎么做——可操作的行动指引}"

**适用范围**: 什么情况下应该想起这条经验

> 标签: `{分类}` `{来源}` `勘误` 或 `补充` 或 `解读`   ← 由 /retro 自动添加
```

新条目**追加到文件顶部**（最新在上），便于快速浏览。

分类标签由 /retro 自动添加：
- 规范：代码规范类
- 质量：代码质量/性能/安全
- 约定：项目约定的写法
- 业务：业务领域知识
- 调试：调试套路/排错技巧

来源标签：
- 纠正：来自纠正捕获
- 复盘：来自标准复盘

经验类型标签（新增）：
- 勘误：参考源说错了，实际行为不同
- 补充：参考源遗漏了关键信息
- 解读：参考源没说错但很费解，需要重新解释

### lessons.idx 索引文件格式

`.specify/memory/lessons.idx` 是一个独立的轻量索引文件，由 `/retro` 自动维护。

格式：每行一条记录，字段用 ` · ` 分隔（空格+中间点+空格）：

```
YYYY-MM-DD · 根因关键词 · 简述
```

设计意图：
- **物理隔离**：索引与正文分离，确保去重扫描成本为 O(n) 且不随经验积累膨胀
- **写入方式**：新经验写入 `lessons.md` 时，同步往 `lessons.idx` 追加一行索引
- **读取方式**：复盘时先只读 `lessons.idx`，命中关键词后才读取 `lessons.md` 对应正文做精确比对

## 调用边界

- **不要**因为"做了一件事"就强行复盘——价值低的工作直接告诉用户"无沉淀价值"
- **不要**绕过用户确认直接写入（纠正捕获模式中非业务类可自动写入除外）
- **不要**把 lessons.md 的内容复制一份到 constitution.md（会成为冗余源）
- **纠正捕获不要过度**：如果只是用户随口一句"这里缩进不对"，不值得走完整流程，直接改了就行
- **宪章修订**：写入 constitution.md 时须标明"待评审"状态，后续通过 `/speckit-constitution` 正式评审
- **去重要严格**：宁可漏掉一条重复经验，也不要让 lessons.md 膨胀
- **当无法判断分类时**：默认归入 lessons.md
